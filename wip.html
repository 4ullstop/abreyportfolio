Scene (Working Title)

  • Platform: PC
  • Software: GCC, Raylib, Visual Studio Code
  • Creation Length: 2 months
  • Link to GitHub Source Code
  • Regular updates on LinkedIn

Scene (Working Title) is a first person puzzler in which your main interaction consists of solving puzzles of buttons which consist of various rules. The other main interaction the player will have is the ability to play between two different windows at once. Through the utilization of multiple executables and shared memory, the player will have to navigate between two different worlds at the same time with challenges spread across both windows.

Responsibilities

  • Develop C code using GCC, Visual Studio Code and Raylib Graphics Library
  • Develop a personalized camera system
  • Develop collision system and collision response code
  • Create a workflow for loading and unloading models from render
  • Develop an interaction system using a custom collision based raycast
  • Develop a system to utilize two windows and shared memory
  • Design and develop button selection puzzle mechanic

Shared Memory

Given the unusual design of this game, I had to devise some interesting techniques to achieve the multi-window gameplay. In order to have two windows communicate with each other, each process needed to open up a portion of shared memory. Then through this memory, items such as the location of the puzzle's cursor or the state of the other game were shared through a struct. The system is fitted to know which puzzle shares with which as well as keep around information such as buttons that have been submitted or where the cursor was when the player exits the puzzle. This code can be found here in the memory_setup.c and sharedpuzzle.c files.

Collision Detection

This project has started with the creation of a custom collision detection and response algorithm which studied and translated from the article Improved Collision detection and Response by Kasper Fauerby. You can view the code for it here in the collisionplane.c and detectcollision.c files.

Handmade Hero

  • Platform: Windows
  • Software: Windows API, DirectXInput, DirectSound
  • GitHub Repo

Handmade Hero is an online series developed by Casey Muratori. With this project, I am developing along side him a 2D adventure game made entirely from scratch. Including things such as writing/copying bitmaps, to writing into the ciruclar sound buffers for DirectSound, and using DirectXInput to capture various input devices. Muratori also provides insight into how to write C++ code in a less Object-Oriented way and provides various techniques into abstracting the various platform layers. My goal is to get to a place where I feel comfortable enough to write a game on my own using the various techniques and strategies gained from the project.

Writing to the Screen

Writing to an open window in Windows requires several operations. We first created a buffer to store our bitmap in. We then write to this buffer every frame, writing to each individual pixel a bitshifted value to create a gradient texture. After which we then copy this buffer into the window using BitBlt.

Writing Sound

Writing to DirectSound is a little more difficult. In order to produce just a sine wave, we have to tell Windows to open a circular sound buffer for us to write into. We then have to compute multiple calculations to ensure we are always writing ahead of our playcursor as well as how to compute the split between the beginning and end of the circular buffer.
